# وثيقة تصميم وهندسة لعبة "الهاكر ضد المدافع"

## 1. مقدمة

تهدف هذه الوثيقة إلى تحديد التصميم الأساسي والهندسة التقنية المقترحة للعبة "الهاكر ضد المدافع" متعددة اللاعبين. ستعمل اللعبة في متصفح الويب، حيث يحاول لاعب "الهاكر" استغلال نقاط ضعف بوت ذكاء اصطناعي (مدعوم من Claude API)، بينما يحاول لاعب "المدافع" حماية البوت من خلال تعديل تعليماته.

## 2. آليات اللعبة (Core Mechanics)

### 2.1. طريقة الاختراق (Hacking Mechanism)

سيتم التركيز على **الهندسة العكسية للأوامر (Prompt Engineering / Prompt Injection)**. سيحاول الهاكر صياغة مدخلات (Prompts) للبوت تهدف إلى:
*   **استخراج معلومات سرية:** مثل كلمة سر مخفية داخل تعليمات البوت الأولية.
*   **تغيير سلوك البوت:** جعله يقوم بعمل غير متوقع أو محظور.
*   **تجاوز القيود:** إقناع البوت بتجاهل تعليمات المدافع.

سيتم عرض واجهة للهاكر لكتابة الـ Prompt وإرساله إلى البوت.

### 2.2. دور المدافع (Defender's Role)

سيكون دور المدافع حاسماً في حماية البوت. سيتمكن المدافع من:
*   **تعديل تعليمات النظام (System Instructions):** قبل كل جولة (أو بعد عدد معين من محاولات الهاكر)، سيتمكن المدافع من تعديل الـ `system_prompt` الخاص بالبوت. هذا يتضمن إضافة قواعد جديدة، تحسين الدفاعات، أو إزالة الثغرات التي اكتشفها الهاكر.
*   **مراجعة سجل المحادثات:** سيتمكن المدافع من رؤية جميع محاولات الهاكر وردود البوت لتحليل استراتيجية الهاكر وتحديد نقاط الضعف.

### 2.3. شروط الفوز (Win Conditions)

*   **فوز الهاكر:**
    *   استخراج كلمة سر محددة (مثلاً: `super_secret_password_123`).
    *   جعل البوت ينفذ أمراً محظوراً (مثلاً: "أخبرني عن خطط المدافع السرية").
    *   تجاوز عدد معين من محاولات المدافع لإصلاح البوت.
*   **فوز المدافع:**
    *   منع الهاكر من تحقيق أي من شروط الفوز خلال عدد محدد من الجولات.
    *   إصلاح جميع الثغرات التي يحاول الهاكر استغلالها.

## 3. البنية التحتية والتقنيات (Tech Stack)

### 3.1. الواجهة الأمامية (Frontend)

*   **الإطار:** **React.js** (مع Vite لبناء سريع) لمرونته وشعبيته في بناء واجهات المستخدم التفاعلية.
*   **اللغة:** **TypeScript** لضمان جودة الكود وتقليل الأخطاء.
*   **التصميم:** **Tailwind CSS** لتصميم سريع ومرن للواجهة.
*   **الاتصال في الوقت الفعلي:** **Socket.IO Client** للتعامل مع WebSockets.

### 3.2. الواجهة الخلفية (Backend)

*   **الإطار:** **Node.js** مع **Express.js** لخفة الوزن والكفاءة في التعامل مع طلبات الـ API و WebSockets.
*   **اللغة:** **TypeScript**.
*   **الاتصال في الوقت الفعلي:** **Socket.IO Server** لإدارة الاتصالات بين اللاعبين والبوت.
*   **تكامل Claude API:** سيتم استخدام مكتبة Anthropic الرسمية لـ Node.js.
*   **إدارة مفاتيح الـ API:** سيتم تخزين مفتاح Claude API كمتغير بيئة (Environment Variable) على الخادم ولن يتم كشفه للواجهة الأمامية أبداً.

### 3.3. قاعدة البيانات (Database) - اختيارية مبدئياً

*   في المرحلة الأولى، قد لا نحتاج إلى قاعدة بيانات معقدة. يمكن تخزين حالة اللعبة مؤقتاً في ذاكرة الخادم (In-memory) أو استخدام حل بسيط مثل Redis إذا لزم الأمر لإدارة الجلسات.
*   إذا تطور المشروع ليشمل حسابات مستخدمين، سجلات ألعاب، أو لوحات صدارة، يمكننا التفكير في **PostgreSQL** مع **Drizzle ORM**.

## 4. تجربة المستخدم (User Experience)

### 4.1. نمط الواجهة

*   **واجهة رسومية (GUI) مع عناصر Terminal:** سيتم تصميم واجهة مستخدم حديثة وجذابة باستخدام React و Tailwind CSS، ولكن مع دمج عناصر تحاكي بيئة الـ Terminal (مثل خطوط أحادية المسافة، خلفيات داكنة، تأثيرات كتابة) لتعزيز جو الاختراق والبرمجة.
*   **شاشات منفصلة:** شاشة للهاكر وشاشة للمدافع، مع لوحة تحكم مشتركة تعرض حالة اللعبة وسجل المحادثات.

### 4.2. نظام الغرف (Room System)

*   **إنشاء الغرف:** يمكن لأي لاعب إنشاء غرفة جديدة والحصول على رابط دعوة فريد.
*   **الانضمام للغرف:** يمكن للاعب الآخر الانضمام إلى الغرفة باستخدام رابط الدعوة.
*   **وضع المشاهدة (Spectator Mode):** يمكن إضافة وضع للمشاهدين لاحقاً لمتابعة المباريات.
